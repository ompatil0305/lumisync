# Campus operating system

## Problem

Campus information is fragmented across maps, directories, dining sites, parking portals, event calendars, departmental pages, and support channels. Students must know which system owns an answer before they can act.

## Product job

Lumisync reduces the effort required to complete a campus task. It should help a user move from intent to a trustworthy next action: navigate, contact, save, verify, or learn more.

## Core jobs to be done

| User intent | Lumisync outcome |
| --- | --- |
| ?Where is my class or professor?? | Find the relevant entity, show location and reliable contact/office details. |
| ?What is open near me?? | Show dining or services with hours, data freshness, and directions. |
| ?Where can I park?? | Explain applicable restrictions and route to a suitable location. |
| ?What can I do today?? | Surface relevant events and organizations with a clear action. |
| ?Can Lumi answer this?? | Provide a cited answer or transparently route to an official source. |

## Experience rules

- Search is an always-available primary path, not a hidden feature.
- Every result provides a next action.
- Missing data is handled with useful alternatives instead of a dead end.
- Personalization is opt-in and should improve a task, not merely add novelty.
- A capability is not shown as real-time unless its source and refresh process support that claim.

## Success measures

- Search success rate and zero-result rate by provider.
- Time from query to meaningful action.
- Navigation completion and route fallback rate.
- Lumi citation coverage, refusal correctness, and user satisfaction.
- Weekly retained students completing meaningful campus tasks.
